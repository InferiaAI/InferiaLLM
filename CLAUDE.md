# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

InferiaLLM is an "operating system" for running LLMs in production. It provides a unified control/data plane for inference routing, access control, safety guardrails, compute orchestration, and observability. Python 3.10-3.12, FastAPI backend, React/TypeScript dashboard.

## Common Commands

```bash
# Setup & Run
make setup                    # Install environment and dependencies
make start                    # Run all services (inferiallm start)
inferiallm init               # Initialize database (first-time setup)
inferiallm start              # Start all microservices

# Testing
make test                     # Run all tests (pytest)
pytest src/tests/api_gateway/test_rbac.py  # Single test file
pytest -k "test_name"         # Run specific test by name

# Docker
make docker-build-unified     # Build unified Docker image
make docker-up-unified        # Start unified stack (app + postgres + redis)
make docker-down              # Stop all services
make docker-clean             # Stop + remove volumes

# Dashboard (React)
cd apps/dashboard && npm run dev    # Dev server with HMR
cd apps/dashboard && npm run build  # Production build → src/dashboard/
cd apps/dashboard && npm run lint   # ESLint

# Package build
python -m build       # Build Python package
```

## Architecture

### Service Layout

All services live as top-level packages under `src/` (`api_gateway/`, `inference/`, `orchestration/`):

| Service | Port | Role |
|---------|------|------|
| `api_gateway` | 8000 | Control plane: auth, RBAC, policy, audit |
| `inference` | 8001 | Data plane: inference request routing |
| `guardrail` | 8002 | Safety scanning, PII detection |
| `data` | 8003 | Knowledge base, RAG operations |
| `orchestration` | 8080 | Compute lifecycle, provider management |
| `filtration` | — | Combined alternative service mode |

### Service Startup Pattern

Each service follows: `main.py` → `start_api()` → `uvicorn.run("app:app")`. The CLI (`inferia.cli`) uses multiprocessing to launch services in parallel with queue-based IPC for status reporting.

### Inter-Service Communication

- **External**: REST/HTTP via FastAPI
- **Internal**: gRPC + protobuf (definitions in `services/orchestration/proto/v1/`)
- **Async tasks**: Redis Streams & Pub/Sub
- **Database**: PostgreSQL via async SQLAlchemy (asyncpg)

### Deployment Modes

- **Unified**: All services in one process/container (default)
- **Split**: Separate container per microservice (via Docker Compose profiles)

### Key Shared Code

- `src/common/` — logging, error schemas, shared utilities
- `src/infra/schema/` — SQL schemas and migration files
- `src/providers/` — pluggable compute provider adapters (aws, nosana, akash, k8s, pulumi, worker)
- `src/cli/` — CLI entry point (package; `cli:main`)
- `src/startup_events.py` — service startup handlers

## Conventions

- **Config**: Pydantic `BaseSettings` with env var injection (per-service `config.py`)
- **Auth**: Stateless JWT + RBAC middleware; API keys for service-to-service
- **Errors**: Standardized `ErrorResponse` model + `APIError` exception class
- **Logging**: JSON structured logging with request ID tracking (stdout; collect with your platform's log driver)
- **Database migrations**: Raw SQL files in `infra/schema/migrations/`
- **Tests**: pytest + pytest-asyncio; fixtures in `conftest.py` provide mock DB sessions and httpx `AsyncClient`
- **Frontend**: React 19 + Vite + TailwindCSS + Shadcn/UI components; TanStack Query for server state

## Working Rules

- Never repeat the same mistake. When a mistake is made, always document it and its edge cases in the "Mistakes Log" section of this file so it is never repeated in future sessions and stays synced with collaborators.
- Refer to official web documentation whenever possible — do not guess at APIs, flags, or behavior.
- Security-first design: validate inputs at system boundaries, avoid command injection in `exec.Command` args, and never expose internal errors to API consumers.
- Design for scalability: prefer approaches that handle growing checkpoint counts, concurrent restores, and large archives without rearchitecting.
- Always use the superpowers plugin for planning and implementing features, debugging, and continuous development.

It is **node-dependent**: whether 803 fires depends on whether `ldconfig` resolves the host libcuda or the compat libcuda FIRST, which depends on where the container runtime injects the host driver (multiarch `/usr/lib/x86_64-linux-gnu` on the AWS DLAMI and some Nosana operators — resolved first, fine; `/usr/lib` on others — compat resolved first, 803). **`NVIDIA_DISABLE_CUDA_COMPAT=1` does NOT fix it** (the compat lib is baked into the ld.so cache at image-build time, not added by the runtime entrypoint hook that env var gates — verified live: the broken container set it and still 803'd). Fix: set a COMPREHENSIVE `LD_LIBRARY_PATH` (`/usr/lib/x86_64-linux-gnu:/usr/lib64:/usr/lib:/lib/x86_64-linux-gnu:/lib64:/lib:/usr/local/nvidia/lib64:/usr/local/cuda/lib64`, constant `CUDA_DRIVER_LD_LIBRARY_PATH`) so the host driver's libcuda is bound wherever the runtime injected it — a superset of the worker's single-dir DLAMI value, robust to any node layout; the worker's `/usr/lib/x86_64-linux-gnu`-only value is NOT enough (e.g. an Arch/CDI host injects to `/usr/lib`). **Bug 2 — `--gpu-memory-utilization 0.95` too aggressive.** vLLM ABORTS at startup if `free_vram < gpu_util * TOTAL` (`ValueError: Free memory on device cuda:0 (10.81/11.62 GiB) ... is less than desired GPU memory utilization (0.95, 11.04 GiB)`). Community GPUs always reserve ~0.8 GiB for the CUDA context (+ co-tenants), so 0.95 (≈95% of total must be free) reliably failed. The dashboard's deploy spec is intentionally MINIMAL (`{model_id, engine, expose, gpu}` — no `gpu_util`), so the adapter's `metadata.get("gpu_util", 0.95)` default was ALWAYS used regardless of the dashboard's `0.80` form field. Fix: default `0.80` in `create_vllm_job`/`create_vllm_omni_job` AND `nosana_adapter.py` (the adapter default is the authoritative one for dashboard deploys). **Diagnosis lessons:** (1) Nosana *confidential* jobs retain NO post-mortem logs (IPFS `opStates: []`) and live-WS container-stdout forwarding is **node-dependent** (some operators forward `[NodeRepository.displayLog]`, some don't) — so to reliably capture a container's failure on a real node, deploy a probe container that tees its output to a file and serves it over the exposed port (`python3 -m http.server`) so you can `curl` it; (2) reproduce the EXACT image+command locally on a GPU with a NEWER driver to force the 803 deterministically (`docker run --gpus all vllm/vllm-openai:<tag> <model> …`), and `find / -name 'libcuda.so*'` + `ldconfig -p | grep libcuda` inside the container to see which lib resolves first; (3) any probe/diagnostic container running `http.server` runs FOREVER → STOP it (`POST /nosana/jobs/stop`) or it bills until the 60-min job timeout. Live-verified end-to-end on real Nosana (RTX-class node, driver 595): CUDA init passes (`device_config=cuda`), `Available KV cache memory: 6.78 GiB`, `Application startup complete`, authenticated `/v1/chat/completions` → real Qwen3-0.6B completion. Fixed in `providers/nosana/job_builder.py` + `providers/nosana/nosana_adapter.py` (branch `feat/unified-provider-deploy-flow`).

- **[2026-06-20] xDS cluster thrashing — `LEFT JOIN` to deployments returned bare nodes with NULL engine, defaulting both to "vllm" and causing cluster name to flip on each poll.** The `inventory_repo.list_xds_nodes()` query used `LEFT JOIN LATERAL` to fetch deployments on each node, returning one row per distinct (engine, model). When a node had NO active deployments (e.g., its only deployment terminated), the LEFT JOIN returned NULL engine/model, which the Python loop defaulted to engine="vllm", model="__default__". This caused **cluster thrashing**: node transitioned from "has ollama deployment" (cluster="grp-pool-ollama") → "no deployments" (cluster="grp-pool-vllm") → xDS hash changed → Envoy received "add 1, remove 1" each poll. Result: Envoy logs `cds: response indicates 1 added/updated cluster(s), 1 removed cluster(s)`, thrashing every DNS/cold-cluster window. Fix: use `INNER JOIN` instead of `LEFT JOIN` so nodes with NO active deployments are excluded entirely (they cannot receive traffic without a deployment, so they don't need a cluster entry). Nodes now only appear in xDS while they have at least one active deployment in state RUNNING/DEPLOYING/PENDING_NODE. Edge cases: (1) a node transitioning from provisioning → ready while having a PENDING_NODE deployment appears immediately (INNER JOIN hits); (2) when a deployment on the node goes FAILED/STOPPED, if it's the last one, the node disappears from xDS on the next poll (correct—cluster is no longer needed); (3) multiple deployments with different engines on the same node still produce multiple xDS rows (one per distinct engine). Regression-guarded: nodes with deployments in all active states appear; nodes with only terminal/inactive deployments don't. Fixed in `src/orchestration/repositories/inventory_repo.py::list_xds_nodes()` (PR #xyz).
