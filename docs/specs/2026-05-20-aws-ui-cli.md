# AWS pool-metadata UI & CLI — design spec

**Date:** 2026-05-20
**Repo:** InferiaLLM
**Status:** approved (small change, spec scope locked)

## Goal

Give operators a way to configure AWS-specific pool metadata
(`subnet_id`, `security_group_ids`, `ami_id`, `iam_instance_profile`,
`root_volume_gb`, `worker_image_tag`) through both the dashboard and CLI.
Today the new `AWSAdapter` reads these from `compute_pools.metadata` jsonb
at provision time, but no UI/CLI surface exists to populate them — pools
created via the dashboard end up with empty `metadata`, and the adapter
fails at `provision_node` with `"pool missing subnet_id or security_group_ids"`.

## Non-goals

- AWS credential CRUD — already exists (`ProviderConfig.tsx → AWSFields`
  and `inferiallm providers add aws`). No change needed.
- Instance-type catalog / live provision buttons / bootstrap-token inspector
  / console-output viewer — explicitly out of scope for this iteration.
- New backend endpoints — `compute_pools.metadata` is jsonb and the existing
  pool PATCH already accepts arbitrary metadata payloads. We add input
  validation only.

## Scope of change

### Backend

1. Add Pydantic validation on the existing pool update payload so AWS pools
   reject invalid `subnet_id`/`security_group_ids` shapes at the boundary
   instead of letting bad data reach `provision_node`. Specifically: when
   the request body contains a `metadata` dict AND the pool provider is
   `"aws"`:
   - `subnet_id` must match `^subnet-[0-9a-f]{8,17}$`
   - `security_group_ids` must be a list of `^sg-[0-9a-f]{8,17}$`, at least one entry
   - `ami_id` (if present) must match `^ami-[0-9a-f]{8,17}$`
   - `iam_instance_profile` (if present) must match `^arn:aws:iam::\d{12}:instance-profile/.+$`
   - `root_volume_gb` (if present) integer, 10 ≤ x ≤ 16384
   - `worker_image_tag` (if present) ≤ 128 chars, no whitespace
   Validation lives in a new Pydantic model `AWSPoolMetadata`; the pool
   update endpoint calls it conditionally.

### Dashboard (React)

2. `apps/dashboard/src/pages/Compute/NewPool.tsx` — add a collapsible
   "AWS provisioning configuration" section that becomes visible when
   `selectedProvider === "aws"`. Fields above. Inline validation against
   the same regex. On submit, bundled into `payload.metadata`.

3. Pool-edit view — investigate whether one exists. If yes, mirror the
   AWS section there. If no, create a minimal `EditPool.tsx` (or
   `PoolDetail.tsx` if one already exists for read-only viewing) with
   just the metadata-edit affordance for AWS pools.

### CLI

4. Extend `package/src/inferia/cli_node.py` to add a `pool` subcommand
   group with a single `aws-config` action initially:
   ```
   inferiallm node pool aws-config POOL_ID \
     --subnet=subnet-abc \
     --security-group=sg-1 [--security-group=sg-2 ...] \
     [--ami=ami-deadbeef] \
     [--iam-profile=arn:aws:iam::123:instance-profile/foo] \
     [--root-gb=200] \
     [--image-tag=v1.2.3]
   ```
   Calls the existing orchestration pool PATCH endpoint, merging into the
   pool's existing metadata (does NOT clobber other keys like
   `provider_pool_id` from Nosana).

5. `inferiallm node pool show POOL_ID` — read-only display. For AWS pools,
   format the metadata table-style; for other providers, print the raw
   metadata jsonb. Lives in the same file.

## Failure modes

| Failure | Surface | Recovery |
|---|---|---|
| Backend validation rejects bad subnet/SG syntax | API 422 | Dashboard shows inline error; CLI prints "invalid subnet_id: must match subnet-..." and exits 1 |
| Pool doesn't exist (CLI) | API 404 | CLI prints "pool not found: <id>" and exits 1 |
| Operator omits subnet/SG when provider=aws (UI) | Form disables Submit | Visual indicator on required-but-empty fields |
| Operator submits metadata with extra/unknown AWS keys | Validation allows (forward-compat) but logs a warning | No-op for the adapter; keys persist but are ignored |
| Pool's provider isn't AWS but operator sends AWS metadata | API 400 `"pool provider is not aws"` | Adapter never sees it |

## Testing

- Backend: `test_pool_aws_metadata_validation.py` covers happy + each rejection case + the non-aws-provider gate. ≥95% coverage on the new `AWSPoolMetadata` model.
- Dashboard: minimal — the React unit-test pattern is rare in this repo (the existing UI work has no per-component unit tests). Verify the AWS section renders/serializes correctly via a quick manual `npm run dev` smoke. Snapshot test optional.
- CLI: `test_cli_pool_aws_config.py` covers the happy path (mocks `urlopen`), each missing-flag rejection, and the merge-not-clobber behavior on existing metadata.

## Open / deferred

- Pool list view that displays AWS metadata at a glance — deferred.
- Bulk pool config import from YAML — deferred.
- Validation of `subnet_id` / `security_group_ids` against the actual AWS
  account at submit time (boto3 lookup) — deferred; saves a round-trip but
  adds a credential dependency on the API call. Today validation is
  syntax-only.
